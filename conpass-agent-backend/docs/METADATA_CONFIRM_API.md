# Metadata Update Confirm API

## Overview
This endpoint executes a metadata update action after the user has approved it. It is called when the user confirms an `UpdateMetadataAction` that was generated by the agent during a chat conversation.

## Endpoint
```
POST /v1/metadata/update
```

## Authentication
- **Required**: Yes
- **Method**: JWT token in request headers (set by authentication middleware)
- The token should be included in the standard authentication header format used by your application

## Request Body

The request body should be the `UpdateMetadataAction` object that was received from the agent. This object contains all the information needed to execute the update.

### Request Schema

```typescript
{
  action_type: "update",
  contract_id: number,              // Required: Target contract ID
  metadata_items: [                  // Required: Array of metadata items to update
    {
      metadata_id: number,           // Required: ID of the metadata record to update
      value?: string,                // Optional: New text value (max 255 chars)
      date_value?: string,           // Optional: New date value (YYYY-MM-DD format)
      lock?: boolean,                // Optional: New lock status (true to lock, false to unlock)
      // Note: At least one of value, date_value, or lock must be provided
    }
  ],
  title: string,                    // Title of the action (for display purposes)
  confirmation_api_url: string,      // URL to call on confirmation (this endpoint)
  cancel_api_url: string,           // URL to call on cancellation
  status: "pending" | "confirmed" | "cancelled",  // Action status
  error_message?: string            // Optional: Error message if validation failed
}
```

### Example Request

```json
{
  "action_type": "update",
  "contract_id": 12345,
  "metadata_items": [
    {
      "metadata_id": 67890,
      "value": "New Company Name",
      "lock": false
    },
    {
      "metadata_id": 67891,
      "date_value": "2024-12-31",
      "lock": true
    }
  ],
  "title": "Update contract metadata",
  "confirmation_api_url": "/v1/metadata/update",
  "cancel_api_url": "/v1/metadata/cancel",
  "status": "pending"
}
```

## Response

### Success Response (200 OK)

```json
{
  "status": "success",
  "message": "Successfully updated 2 metadata item(s)",
  "data": {
    // Response data from ConPass API (structure may vary)
  }
}
```

### Error Response (200 OK with error status)

The API returns HTTP 200 even on errors, but includes an error status in the response body:

```json
{
  "status": "error",
  "message": "Failed to update metadata",
  "error_details": "Detailed error message from ConPass API",
  "data": {
    // May contain additional error data
  }
}
```

### Validation Error Response

If the request is invalid (e.g., missing metadata_id, no metadata items):

```json
{
  "status": "error",
  "message": "No valid metadata items to update",
  "error_details": "All metadata items are missing required metadata_id"
}
```

### Authentication Error (401 Unauthorized)

```json
{
  "detail": "Authentication required"
}
```

## Usage Flow

1. **Agent generates action**: During a chat conversation, the agent may generate an `UpdateMetadataAction` object when the user requests a metadata update.

2. **Frontend displays confirmation**: The frontend should display the action details to the user for approval, showing:
   - The contract ID
   - The metadata items that will be updated
   - Current vs. new values (if available in the action object)

3. **User confirms**: When the user clicks "Confirm" or "Approve":
   - Send a POST request to `/v1/metadata/update` with the entire `UpdateMetadataAction` object as the request body
   - Include authentication headers as required

4. **Handle response**: 
   - If `status === "success"`: Show success message to user
   - If `status === "error"`: Display the error message and `error_details` to the user

## Important Notes

- **All metadata items must have `metadata_id`**: The `metadata_id` field is required for all items in the `metadata_items` array
- **At least one field must be updated**: Each metadata item should have at least one of: `value`, `date_value`, or `lock`
- **Date format**: `date_value` must be in `YYYY-MM-DD` format (e.g., "2024-12-31")
- **Value length**: `value` field has a maximum length of 255 characters
- **Person metadata**: For person metadata (when `key.label === 'conpass_person'`), the `value` should be comma-separated person IDs (e.g., "1,2,3")
- **Empty arrays**: The request will fail if `metadata_items` is empty or if all items are invalid

## Example Frontend Implementation

```typescript
async function confirmMetadataUpdate(action: UpdateMetadataAction) {
  try {
    const response = await fetch('/v1/metadata/update', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        // Include your authentication header here
        'Authorization': `Bearer ${authToken}`
      },
      body: JSON.stringify(action)
    });

    const result = await response.json();

    if (result.status === 'success') {
      // Show success message
      console.log('Metadata updated successfully:', result.message);
      return result;
    } else {
      // Show error message
      console.error('Update failed:', result.message, result.error_details);
      throw new Error(result.error_details || result.message);
    }
  } catch (error) {
    console.error('Request failed:', error);
    throw error;
  }
}
```
