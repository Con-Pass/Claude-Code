# GCS and Cloud CDN Configuration Guide

This guide documents the Google Cloud Storage (GCS) and Cloud CDN configuration for the ConPass Agent backend file upload system.

## Architecture Overview

```text
User Request
    |
    v
Cloud CDN (www.xyz.con-pass.jp)
    |
    v
HTTPS Load Balancer
    |
    v
Backend Bucket (GCS)
    |
    v
Private GCS Bucket (conpass-agent-bucket-dev)
```

**Key Points:**

- GCS bucket is **private** (not publicly accessible)
- Files are accessed via **signed URLs** generated by the application
- Cloud CDN caches and serves files through the load balancer
- HTTPS is enforced with SSL certificates

## Prerequisites

1. **Google Cloud Project**: `conpass-agent`
2. **GCS Bucket**: `conpass-agent-bucket-dev` (must exist)
3. **Domain**: `www.xyz.con-pass.jp` (must be owned and DNS accessible)
4. **gcloud CLI**: Configured and authenticated

## Step 1: GCS Bucket Setup

### 1.1 Create GCS Bucket (if not already created)

```bash
gsutil mb -p conpass-agent -l asia-northeast1 gs://conpass-agent-bucket-dev
```

Or using the console:

1. Go to [Cloud Storage Console](https://console.cloud.google.com/storage)
2. Click **CREATE BUCKET**
3. Name: `conpass-agent-bucket-dev`
4. Location type: `Region` (e.g., `asia-northeast1`)
5. Storage class: `Standard`
6. Access control: `Uniform`
7. **Important**: Keep bucket **private** (do not enable public access)

### 1.2 Configure Bucket Permissions

Grant your service account (or Compute Engine default service account) access to the bucket:

```bash
# Get your service account email (usually ends with @compute.gserviceaccount.com)
gcloud iam service-accounts list

# Grant Storage Object Admin role (or appropriate permissions)
gsutil iam ch serviceAccount:YOUR_SERVICE_ACCOUNT@conpass-agent.iam.gserviceaccount.com:roles/storage.objectAdmin gs://conpass-agent-bucket-dev
```

**Required IAM Roles:**

- `roles/storage.objectCreator` - To upload files
- `roles/storage.objectViewer` - To read files (for signed URLs)
- Or `roles/storage.objectAdmin` - Full object access

## Step 2: Create Backend Bucket

The backend bucket links your GCS bucket to the load balancer:

```bash
gcloud compute backend-buckets create conpass-agent-backend-bucket \
  --gcs-bucket-name=conpass-agent-bucket-dev \
  --enable-cdn
```

**Note**: The `--enable-cdn` flag enables Cloud CDN caching for this backend.

## Step 3: Create URL Map

The URL map routes requests to the backend bucket:

```bash
gcloud compute url-maps create conpass-agent-cdn-map \
  --default-backend-bucket=conpass-agent-backend-bucket
```

This creates a simple URL map that routes all requests to your GCS backend bucket.

## Step 4: Create SSL Certificate

Create a Google-managed SSL certificate for your domain:

```bash
gcloud compute ssl-certificates create conpass-agent-ssl-cert \
  --domains=www.xyz.con-pass.jp
```

**Important Notes:**

- The certificate will be in `PROVISIONING` state initially
- Google will provision the certificate (may take 10-60 minutes)
- You must have DNS configured before the certificate can be provisioned
- Check status with: `gcloud compute ssl-certificates describe conpass-agent-ssl-cert`

**Certificate Status:**

```text
NAME: conpass-agent-ssl-cert
TYPE: MANAGED
CREATION_TIMESTAMP: 2026-01-04T21:23:20.679-08:00
REGION: 
MANAGED_STATUS: PROVISIONING (or ACTIVE when ready)

www.xyz.con-pass.jp: PROVISIONING (or ACTIVE when ready)
```

## Step 5: Create Target HTTPS Proxy

Create a target HTTPS proxy that links the URL map to the SSL certificate:

```bash
gcloud compute target-https-proxies create conpass-agent-https-proxy \
  --url-map=conpass-agent-cdn-map \
  --ssl-certificates=conpass-agent-ssl-cert
```

This creates:

```text
NAME: conpass-agent-https-proxy
SSL_CERTIFICATES: conpass-agent-ssl-cert
URL_MAP: conpass-agent-cdn-map
```

## Step 6: Reserve Static IP Address

Reserve a global static IP address for the load balancer:

```bash
gcloud compute addresses create conpass-agent-cdn-ip \
  --global
```

Get the IP address:

```bash
gcloud compute addresses describe conpass-agent-cdn-ip --global
```

**Output Example:**

```text
address: 34.102.136.180
addressType: EXTERNAL
kind: compute#address
name: conpass-agent-cdn-ip
networkTier: PREMIUM
selfLink: https://www.googleapis.com/compute/v1/projects/conpass-agent/global/addresses/conpass-agent-cdn-ip
status: RESERVED
```

## Step 7: Create Forwarding Rule

Create a global forwarding rule to route HTTPS traffic to the proxy:

```bash
gcloud compute forwarding-rules create conpass-agent-https-rule \
  --global \
  --target-https-proxy=conpass-agent-https-proxy \
  --ports=443 \
  --address=conpass-agent-cdn-ip
```

This creates:

```text
NAME: conpass-agent-https-rule
TARGET: conpass-agent-https-proxy
IP_ADDRESS: [reserved IP address]
PORT_RANGE: 443-443
```

## Step 8: Configure DNS

Configure your DNS provider to point your domain to the reserved IP address:

1. **Go to your DNS provider** (where you manage `xyz.con-pass.jp`)
2. **Create an A Record**:
   - **Type**: A
   - **Name**: `www` (or `@` for root domain)
   - **Value**: `34.102.136.180` (your reserved IP address from Step 6)
   - **TTL**: 3600 (or your preference)

**DNS Configuration Example:**

```text
Type: A
Name: www
Value: 34.102.136.180
TTL: 3600
```

**Verify DNS propagation:**

```bash
dig www.xyz.con-pass.jp
# or
nslookup www.xyz.con-pass.jp
```

The A record should point to your reserved IP address.

## Step 9: Wait for SSL Certificate Provisioning

After DNS is configured, wait for Google to provision the SSL certificate:

```bash
# Check certificate status
gcloud compute ssl-certificates describe conpass-agent-ssl-cert
```

The certificate status should change from `PROVISIONING` to `ACTIVE`. This typically takes:

- **10-60 minutes** after DNS is correctly configured
- May take longer if DNS hasn't propagated globally

**Do not proceed until the certificate is ACTIVE.**

## Step 10: Application Configuration

Update your application environment variables:

### .env File

```bash
# GCS Configuration
GCS_BUCKET_NAME=conpass-agent-bucket-dev
CDN_DOMAIN=www.xyz.con-pass.jp
```

**Important Notes:**

- `CDN_DOMAIN` should be **just the domain** (e.g., `www.xyz.con-pass.jp`)
- **Do not include** `https://` or path prefixes like `/files`
- The application will generate signed GCS URLs (not CDN URLs) due to signature validation requirements

### Application Code Behavior

The application uses **GCS signed URLs** (not CDN URLs) because:

- GCS v4 signed URLs are hostname-specific
- The signature cannot be validated if the hostname is changed
- Cloud CDN will forward requests to GCS, which validates the signature

**URL Format:**

```text
https://storage.googleapis.com/conpass-agent-bucket-dev/files/{file_id}/{filename}?X-Goog-Algorithm=...&X-Goog-Credential=...&X-Goog-Signature=...
```

**Note:** Even though Cloud CDN is configured, the application returns GCS signed URLs directly. The CDN can still cache and serve files, but the URL will show the GCS domain.

## Troubleshooting

### SSL Certificate Not Provisioning

**Symptoms:** Certificate stays in `PROVISIONING` state

**Solutions:**

1. Verify DNS A record points to the correct IP
2. Check DNS propagation: `dig www.xyz.con-pass.jp`
3. Ensure DNS TTL allows updates
4. Wait longer (can take up to 60 minutes)
5. Check Google Cloud Console for error messages

### 403 Forbidden Errors

**Symptoms:** `Access denied` or `403 Forbidden` when accessing files

**Possible Causes:**

1. **Bucket is private** - This is expected. Files should be accessed via signed URLs
2. **Signed URL signature mismatch** - Check if URL was modified
3. **Service account permissions** - Verify IAM roles are correct
4. **Signed URL expired** - Default expiration is 30 minutes

### Signed URLs Not Working

**Symptoms:** `SignatureDoesNotMatch` error

**Solutions:**

- Ensure you're using GCS signed URLs (not attempting to replace hostname)
- Check that the signature hasn't expired
- Verify service account has proper permissions
- Ensure the blob path is correct

### CDN Not Caching

**Symptoms:** No caching headers, high latency

**Solutions:**

1. Verify `--enable-cdn` flag was used when creating backend bucket
2. Check cache-control headers on objects
3. Ensure requests are going through the load balancer (not direct GCS access)
4. Review Cloud CDN cache configuration

## Architecture Notes

### Signed URLs and Cloud CDN

**Current Implementation:**

- Application generates **GCS signed URLs** (using GCS hostname)
- URLs are returned to clients
- Clients access files via GCS signed URLs
- Cloud CDN can still cache if requests go through the load balancer

**Why not CDN domain URLs with signatures?**

- GCS v4 signed URLs are hostname-specific
- The signature validation fails if the hostname is changed
- Cannot replace `storage.googleapis.com` with `www.xyz.con-pass.jp` in signed URLs

**Alternative Approaches (for future consideration):**

1. **Public objects with CDN**: Make objects public, serve via CDN (less secure)
2. **Cloud CDN signed URLs**: Use Cloud CDN's own signed URL mechanism (different setup)
3. **IAM-based access**: Use IAM authentication with Cloud CDN (requires different architecture)

### Security Considerations

1. **Bucket is private**: Objects are not publicly accessible
2. **Signed URLs**: Time-limited access (default 30 minutes)
3. **HTTPS only**: All traffic encrypted
4. **Service account permissions**: Limited IAM roles for service account
5. **No public access**: Bucket does not allow anonymous access
