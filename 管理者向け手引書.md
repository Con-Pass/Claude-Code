## 1. 管理者ログインと管理画面

### 管理者アカウントの種類

| タイプ | 権限 |
|-------|------|
| **ACCOUNT管理者** | テナント内のユーザー・ディレクトリ・設定を管理 |
| **ADMIN（日本パープル）** | 全テナントの管理・システム全体の操作 |

### Django 管理画面へのアクセス

```
URL: /admin/
ユーザー: ADMIN タイプのユーザーアカウント
```

Django 管理画面（`/admin/`）では、すべてのモデルに対して直接 CRUD 操作が可能です。通常の運用では使用せず、緊急時のデータ修正や初期設定のみに使用してください。

### APIによる管理操作

管理者向けの主要 API エンドポイント：

```
GET  /api/user/all                   # ユーザー一覧
POST /api/user/permission/set        # 権限設定
GET  /api/setting/directory/all      # ディレクトリ一覧
POST /api/setting/directory/update   # ディレクトリ更新
GET  /api/setting/meta               # メタキー一覧
POST /api/setting/meta/update        # メタキー更新
POST /api/setting/notification/edit  # 通知設定変更
```

## 2. ユーザー管理

### ユーザーの追加

#### 個別追加

```
POST /api/auth/signup（または管理画面から直接登録）

必要なフィールド:
- login_name（メールアドレス）
- username（表示名）
- division（部署名）
- position（役職）
- type: 1=ACCOUNT / 2=CLIENT / 3=ADMIN
- account（所属テナント）
```

#### CSV一括アップロード

1. 所定のCSVフォーマット（ヘッダー：`login_name,username,division,...`）でユーザーリストを作成します。
2. `POST /api/user/upload` に CSV ファイルをアップロードします。
3. 処理結果（成功件数・エラー件数）がレスポンスで返ります。

> **CSV フォーマット**: `POST /api/user/data/list` で既存ユーザーデータをダウンロードし、それをテンプレートとして使用してください。

### ユーザーの権限設定

```
POST /api/user/permission/set

{
  "user_id": 123,
  "permissions": [
    { "target": "contract_read", "is_allow": true },
    { "target": "contract_write", "is_allow": true },
    { "target": "contract_delete", "is_allow": false }
  ]
}
```

### BPOユーザーの設定

BPO（業務委託）モードのユーザーは、通常ユーザーとは別の操作フロー（メタデータ修正依頼の受付・対応）を使用します。

```python
# Django管理画面またはAPIで設定
user.is_bpo = True
user.save()
```

## 3. グループと権限の設定

### グループ管理

グループを使うと、複数ユーザーに一括で権限を付与できます。

```
GET  /api/group/all       # グループ一覧取得
POST /api/group/edit      # グループ作成・更新

{
  "name": "法務部",
  "users": [1, 2, 3],       # ユーザーIDリスト
  "permissions": [...]       # 権限リスト
}
```

### ディレクトリ権限（DirectoryPermission）

ディレクトリごとにアクセス権を設定できます。

| 権限 | 内容 |
|-----|------|
| **読み取り（read）** | 契約書の閲覧・検索 |
| **作成（create）** | 新規契約書のアップロード |
| **更新（update）** | メタ情報・コメントの編集 |
| **削除（delete）** | 契約書のゴミ箱移動 |

設定方法：

```python
# Django管理画面 > DirectoryPermission
DirectoryPermission.objects.create(
    directory=directory,
    group=group,          # グループに付与
    # または
    user=user,            # 個人に付与
    can_read=True,
    can_create=True,
    can_update=True,
    can_delete=False,
)
```

### 権限カテゴリ（PermissionCategory）

よく使う権限セットを「カテゴリ」としてまとめておくと、新規ユーザーへの権限付与が効率化されます。

## 4. ディレクトリ（フォルダ）管理

### ディレクトリの構造

ディレクトリは階層構造（フォルダツリー）で契約書を整理します。

```
例:
├── 取引先別
│   ├── A社
│   │   ├── 基本契約
│   │   └── 個別契約
│   └── B社
├── 部門別
│   ├── 営業部
│   └── 技術部
└── アーカイブ
```

### ディレクトリの作成・編集

```
POST /api/setting/directory/update

{
  "name": "A社",
  "parent_id": 5,     # 親ディレクトリID（最上位はnull）
  "type": "CONTRACT", # CONTRACT / TEMPLATE / PAST
  "memo": "2020年より取引開始"
}
```

### ディレクトリ削除

```
POST /api/setting/directory/delete
{ "directory_id": 10 }
```

> **注意**: 契約書が存在するディレクトリは削除できません。先に契約書を移動またはアーカイブしてください。

### ディレクトリとメタキーの紐付け

ディレクトリごとに「このディレクトリで使用するメタ項目」を定義できます（MetaKeyDirectory）。

例：
- **法務ディレクトリ** → 「契約種別」「管轄裁判所」「損害賠償上限額」を表示
- **HR ディレクトリ** → 「雇用形態」「入社日」「退職日」を表示

```
GET  /api/setting/meta/directory   # ディレクトリのメタキー設定取得
POST /api/setting/directory/meta/update  # 設定更新
```

### ソート順の変更

```
POST /api/setting/directory/sort/edit
{ "sorted_directory_ids": [3, 1, 5, 2, 4] }
```

## 5. メタキー設定

メタキーは契約書に付与できる「属性項目」の定義です。

### 標準メタキー（変更不可）

| ラベル（DB） | 表示名 | 型 |
|------------|------|---|
| title | 契約書名 | TEXT |
| companya | 会社名（甲） | TEXT |
| companyb | 会社名（乙） | TEXT |
| contractdate | 契約日 | DATE |
| contractstartdate | 契約開始日 | DATE |
| contractenddate | 契約終了日 | DATE |
| conpass_contract_type | 契約種別 | TEXT |
| cort | 管轄裁判所 | TEXT |

### カスタムメタキーの追加

テナント独自の項目を追加できます。

```
POST /api/setting/meta/update

{
  "name": "契約金額",
  "input_type": "TEXT",  # TEXT / DATE / CHOICE / MULTI_SELECT
  "required": false,
  "choice_list": null     # CHOICE型の場合: ["選択肢1", "選択肢2"]
}
```

### メタキーの有効/無効

使わなくなったメタキーは削除せず無効化することを推奨します（既存データへの影響を避けるため）。

## 6. 通知設定

### 通知タイプと設定

```
POST /api/setting/notification/edit

{
  "type": 2,           # 1=WORKFLOW / 2=CONTRACT_DEADLINE / 3=CONTRACT_MONTH
  "days_before": 30,   # 期限の何日前に通知するか
  "info": true,        # アプリ内通知
  "mail": true         # メール通知
}
```

### 期限通知の推奨設定

| 通知 | 推奨タイミング |
|-----|-------------|
| 1回目 | 期限90日前 |
| 2回目 | 期限30日前 |
| 3回目 | 期限7日前 |

複数の通知ルールを作成することで、段階的なリマインダーが設定できます。

### メールテンプレート（MailTemplate）

通知メールの文面はカスタマイズできます。

```python
# Django管理画面 > MailTemplate
# 利用可能なタグ変数例:
# {{ contract_name }}  - 契約書名
# {{ deadline }}       - 期限日
# {{ user_name }}      - 通知先ユーザー名
# {{ contract_url }}   - 契約書詳細URLリンク
```

## 7. ワークフローテンプレートの管理

### テンプレートの種類

| 種別 | 説明 | 編集可否 |
|-----|------|--------|
| **SYSTEM_TEMPLATE** | 日本パープルが提供する標準テンプレート | 不可（コピーのみ） |
| **TEMPLATE** | テナントが作成したカスタムテンプレート | 可 |
| **WORKFLOW** | 実際に進行中のワークフローインスタンス | 不可 |

### ワークフローテンプレートの作成

```
POST /api/workflow/edit

{
  "name": "標準承認フロー",
  "type": 2,   # TEMPLATE
  "params": {
    "steps": [
      {
        "name": "担当者確認",
        "tasks": [{ "user_id": 10 }]
      },
      {
        "name": "法務チェック",
        "tasks": [{ "group_id": 3 }]
      },
      {
        "name": "部長承認",
        "tasks": [{ "user_id": 5 }]
      }
    ]
  }
}
```

### ステップ設定のポイント

- **並列承認**: 同一ステップに複数のタスクを設定すると、全員の承認が必要になります。
- **グループ割り当て**: `group_id` を指定するとグループ内のいずれか1名が承認すれば次のステップへ進みます。
- **コメント必須**: ステップに `require_comment: true` を設定すると、承認時にコメント入力が必須になります。

## 8. Playbookの設定

Playbook は契約書の条項ポリシーを定義し、AIが自動で契約書をレビュー・評価する機能です。

### Playbookの作成

1. テンプレートの選択（業種別：IT / 建設 / 医療 / 汎用など）
2. テナント用にカスタマイズ

```
POST /api/tenant/playbook/create

{
  "name": "標準SaaS契約Playbook",
  "your_side": "VENDOR",    # VENDOR / CUSTOMER / LICENSOR / LICENSEE
  "template_id": 3
}

POST /api/tenant/playbook/<id>/apply-template
{ "template_id": 3 }
```

### 条項ポリシー（ClausePolicy）の設定

各条項タイプに対して「理想条件（GREEN）」「交渉可能範囲（YELLOW）」「エスカレーション条件（RED）」を定義します。

| 条項タイプ | 例 |
|----------|---|
| LIABILITY | 損害賠償条項 |
| INDEMNIFICATION | 補償条項 |
| CONFIDENTIALITY | 秘密保持条項 |
| IP | 知的財産権条項 |
| TERMINATION | 解除条項 |
| WARRANTY | 保証条項 |
| DATA_PROTECTION | データ保護条項 |

```json
// ClausePolicy 例（損害賠償）
{
  "clause_type": "LIABILITY",
  "standard_position": {
    "description": "年間契約料の12ヶ月分を上限とする",
    "cap": "annual_fee_12x"
  },
  "acceptable_range": {
    "min_cap": "annual_fee_6x",
    "max_cap": "annual_fee_24x"
  },
  "escalation_triggers": {
    "conditions": ["無制限損害賠償", "連帯保証"]
  }
}
```

### ルール（TenantRule）の設定

```
GET /api/tenant/rules         # ルール一覧
GET /api/tenant/rules/alerts  # アラート一覧

// ルール例: 契約終了日が30日以内の場合にアラート
{
  "rule_type": "DEADLINE",
  "condition": { "days_before_end": 30 },
  "action": "ALERT",
  "priority": 1
}
```

## 11. Qdrantインジェスト（AIインデックス）

AIによるセマンティック検索を機能させるには、MySQL の契約書データを Qdrant ベクトルDB にインデックスする必要があります。

### 全件インジェスト

```bash
docker exec conpass-agent sh -c 'cd /app && uv run scripts/local_ingest.py'
```

### 特定の契約書のみ再インジェスト

```bash
docker exec conpass-agent sh -c 'cd /app && uv run scripts/local_ingest.py --contract-id 30'
```

### 件数を限定してテスト

```bash
docker exec conpass-agent sh -c 'cd /app && uv run scripts/local_ingest.py --limit 5'
```

### インジェスト処理の内容

実行すると以下の処理が順に行われます：

1. **Qdrantペイロードインデックスの確認・作成**
   - `contract_category`, `contract_type`, `contract_subtype` など新フィールドのインデックスを自動作成
2. **MySQL から契約書データ取得**
   - 有効な契約書（status が有効、ゴミ箱でない）のみ対象
   - 本文テキスト（ContractBody）とメタデータを取得
3. **AI による契約種別分類**
   - タイトル・本文をもとにルールベース + GPT-4o-mini で3階層分類
   - `contract_category`（大分類）/ `contract_type`（中分類）/ `contract_subtype`（小分類）
4. **Related contract の解決**
   - メタデータの `related_contract`（タイトル文字列）を実際の契約書IDに解決
5. **ベクトル生成と書き込み**
   - Dense（384次元）+ Sparse（BM25）のハイブリッドベクトルを生成
   - Qdrant にアップサート

### インジェスト対象の条件

MySQL の `conpass_contract` テーブルで以下の条件を満たす契約書が対象です。

```sql
type = 1
AND status NOT IN (0, 10, 11, 20)  -- 無効・ゴミ箱などを除外
AND is_garbage = 0
```

本文（ContractBody）が存在しない契約書はスキップされます。

### Qdrant コレクションの再作成

Embedding モデルの変更（次元数が変わる場合）は、コレクションを作り直す必要があります。

```bash
# Qdrant Cloud の REST API でコレクション削除・再作成
curl -X DELETE "https://<qdrant-url>/collections/conpass" \
  -H "api-key: <api-key>"

curl -X PUT "https://<qdrant-url>/collections/conpass" \
  -H "api-key: <api-key>" \
  -H "Content-Type: application/json" \
  -d '{
    "vectors": {
      "dense": {
        "size": 384,
        "distance": "Cosine"
      }
    },
    "sparse_vectors": {
      "sparse": { "index": { "on_disk": false } }
    }
  }'
```

その後、全件インジェストを実行してください。

### ペイロードインデックスの確認

Qdrant Cloud のコンソール、または以下の API でインデックス一覧を確認できます。

```bash
curl "https://<qdrant-url>/collections/conpass" \
  -H "api-key: <api-key>" | jq '.result.payload_schema | keys'
```

インデックスが存在しないフィールドをフィルタに使うと 400 エラーが発生します。その場合はインジェストスクリプトを再実行すると `ensure_payload_indexes()` が自動作成します。

## 12. トラブルシューティング

### AIチャットが 401 エラーを返す

**原因**: ユーザーの `auth-token` Cookie が期限切れ（セッション切れ）。

**対処**: ユーザーに再ログインを依頼してください。セッション有効期間は Django の `SESSION_COOKIE_AGE` で設定されています。

### AIチャットで「該当する契約が見つかりません」

**考えられる原因**:
1. Qdrant にまだインデックスされていない → インジェストスクリプトを実行
2. 対象契約書がユーザーのアクセス可能ディレクトリにない → DirectoryPermission を確認
3. Qdrant 検索でスコアが閾値以下 → `semantic_search_tool.py` の `SCORE_THRESHOLD` を調整

### Qdrant 400 エラー（"Not existing vector name error"）

**原因**: クエリが名前付きベクトル（`dense`）の指定なしで送信されている。

**対処**: `semantic_search_tool.py` の fallback クエリに `"using": "dense"` が含まれているか確認してください。

### OCRのテキスト抽出精度が低い

**対処**:
1. 元のPDFが画像PDFか（スキャンPDF）か確認 → 画像PDFは Vision API OCR で処理
2. Google Vision API のエラーログを確認: `docker logs conpass-backend-web-1 | grep Vision`
3. 「🔄 再抽出」ボタンで再処理を試行
4. テキストPDFの場合は pdfminer.six による直接抽出が使われるため精度が高い

### メール通知が届かない

1. SendGrid のダッシュボードで配信ログを確認
2. スパムフォルダを確認
3. `.env` の `SENDGRID_API_KEY` が正しいか確認
4. 通知設定（`NotificationSetting`）で `mail: true` になっているか確認

### コンテナ起動後にモデルダウンロードが遅い

FastEmbed モデル（`sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`）は初回起動時にダウンロードされます。コンテナを再起動するたびにダウンロードが発生する場合は、Docker ボリュームにモデルキャッシュを保存する設定を検討してください。

```yaml
# docker-compose.yml に追加
volumes:
  - fastembed_cache:/root/.cache/fastembed

volumes:
  fastembed_cache:
```

### Qdrant コレクションのポイント数が期待より少ない

```bash
# インジェストログを確認
docker exec conpass-agent sh -c 'cd /app && uv run scripts/local_ingest.py 2>&1' | grep -E "(スキップ|書き込み完了|インデックス完了)"
```

スキップ件数が多い場合は、対象契約書に ContractBody（本文データ）が存在しないことを意味します。OCR 処理が完了しているか `conpass_contractbody` テーブルを確認してください。
