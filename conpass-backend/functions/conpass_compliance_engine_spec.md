# ConPass コンプライアンス判定エンジン改修指示書

## 背景と目的

ConPassのAIコンプライアンスチェック機能のPoCにおいて、以下の深刻な誤判定が発生した。

**発生した事象：**
- 契約タイトル「産業廃棄物処理委託契約書」→「農業経営に関する契約書」と誤認識
- 当事者「株式会社日本パープル」→「株式会社ポーブル」と誤認識
- 実際には記載されている法定事項（マニフェスト、許可証、廃棄物種類等）を「記載なし」と誤判定
- リスクレベルHIGHと判定されたが、実際の契約書は廃掃法上かなり整備された内容

**根本原因：** OCR/PDF解析の精度不足、およびプロンプトの汎用化に伴う判定精度の低下

本指示書では、以下の優先度で改修を実施する。

| 優先度 | 対策 | 効果 |
|--------|------|------|
| P0 | OCR/PDF解析の精度検証と改善 | 誤判定の根本原因を解消 |
| P1-A | 読み取り不能時のUNREADABLE判定を追加 | 偽陽性の大幅削減 |
| P1-B | 法令ごとのチェックポイントをarticles_blockに含める | 汎用化による精度低下を防止 |
| P1-C | チェックポイントJSONをRAG登録済み条文で検証する | チェックポイント自体の誤りを排除 |
| P2 | 別表の存在確認ロジックをパイプライン側に追加 | 偽陰性の削減 |

---

## P0: OCR/PDF解析の精度検証と改善

### 課題

`{contract_text}`に渡されるテキストが不正確な場合、プロンプトをいくら改善しても正しい判定はできない。現状のPDF解析パイプラインで以下の問題が確認されている。

- 固有名詞の誤認識（パープル→ポーブル）
- 契約タイトルの完全な誤読
- 別表・附表のテーブル構造の読み落とし

### 実装指示

1. **PDF解析ライブラリの検証**
   - 現在使用しているPDF解析ライブラリ（pymupdf, pdfplumber等）の出力を、実際の契約書PDFに対して検証すること
   - 特に日本語フォント（游明朝、游ゴシック、MS明朝等）の認識精度を確認
   - テーブル構造（別表1〜4）が正しくテキスト化されているか確認

2. **解析品質スコアの導入**
   - PDF解析後に、以下の観点で品質スコアを算出するロジックを追加する
     - テキスト長が妥当か（契約書なら通常1000文字以上）
     - 日本語として意味が通るか（文字化け率の算出）
     - 契約書の基本構造（条文番号「第○条」等）が検出できるか
   - 品質スコアが閾値以下の場合、LLM判定に進まずUNREADABLEを返す

3. **別表テキストの結合**
   - PDF内の別表・附表が本文とは別ページにある場合でも、`{contract_text}`に結合して渡すこと
   - 結合時に `【本文】` `【別表1】` `【別表2】` 等のセクション区切りを付加する

## P1-A: UNREADABLE判定の追加

### 課題

現在のプロンプトに「契約書テキストが非常に短く判断困難な場合は MEDIUM としてください」というルールがあるが、OCR失敗で空テキスト・断片的テキストになった場合にすべてMEDIUMになり、偽陽性が大量発生する。

### 実装指示

1. **LLM呼び出し前の前処理として、以下のガードを追加する**

```python
def pre_validate_contract_text(contract_text: str) -> dict:
    """
    LLM呼び出し前に contract_text の品質を検証する。
    品質不足の場合は UNREADABLE を返し、LLM呼び出しをスキップする。
    """
    # ガード1: テキスト長
    if len(contract_text.strip()) < 100:
        return {
            "risk_level": "UNREADABLE",
            "reason": "契約書テキストが100文字未満です。PDF解析が正常に完了していない可能性があります。",
            "gaps": []
        }
    
    # ガード2: 契約書構造の検出
    structure_markers = ["第1条", "第１条", "第一条", "条項", "契約"]
    has_structure = any(marker in contract_text for marker in structure_markers)
    if not has_structure:
        return {
            "risk_level": "UNREADABLE",
            "reason": "契約書の基本構造（条文番号等）が検出できません。対象文書が契約書でないか、PDF解析に失敗している可能性があります。",
            "gaps": []
        }
    
    # ガード3: 文字化け率
    # 制御文字や意味不明な文字の割合をチェック
    garbage_chars = sum(1 for c in contract_text if ord(c) < 32 and c not in '\n\r\t')
    garbage_ratio = garbage_chars / len(contract_text) if contract_text else 1.0
    if garbage_ratio > 0.1:
        return {
            "risk_level": "UNREADABLE",
            "reason": f"テキストの文字化け率が{garbage_ratio:.0%}と高く、PDF解析が正常に完了していない可能性があります。",
            "gaps": []
        }
    
    return None  # 検証OK、LLM判定に進む
```

2. **プロンプトからも「短すぎて判断不能→MEDIUM」ルールを変更する**

```
# 旧ルール（削除）
- 契約書テキストが非常に短く判断困難な場合は MEDIUM としてください

# 新ルール（追加）
- 契約書テキストが断片的で構造を認識できない場合は UNREADABLE を返してください
- UNREADABLE の場合は gaps にテキスト品質の問題点を記載してください
```

3. **判定結果のスキーマにUNREADABLEを追加する**

```python
VALID_RISK_LEVELS = ["COMPLIANT", "LOW", "MEDIUM", "HIGH", "UNREADABLE"]
```

4. **チェックポイント未設定時のフォールバックモード判定**

```python
def determine_evaluation_mode(law_config: dict) -> str:
    """
    チェックポイントの有無に応じて、判定モードを決定する。
    
    Returns:
        "checkpoint": 通常モード（チェックポイント＋条文テキスト）
        "fallback":   フォールバックモード（条文テキストのみのCoT）
    """
    if law_config and law_config.get("checkpoints"):
        return "checkpoint"
    else:
        return "fallback"
```

フォールバックモードの場合:
- プロンプトをフォールバック専用のものに切り替える（後述）
- 判定結果に `"mode": "fallback"` フラグを付加する
- UI側で「⚠ チェックポイント未設定のため、条文ベースの判定です。精度が通常より低下する可能性があります。」と表示する

```python
def evaluate_contract_compliance(contract_text, law_config, articles_text, ...):
    # ... pre_validate_contract_text() ...
    
    mode = determine_evaluation_mode(law_config)
    
    if mode == "checkpoint":
        prompt = build_checkpoint_prompt(law_config, articles_text, contract_text, ...)
    else:
        prompt = build_fallback_prompt(law_config, articles_text, contract_text, ...)
    
    result = call_llm(prompt)
    result["mode"] = mode  # 判定モードを結果に付加
    
    return result
```

## P1-B: 法令ごとのチェックポイント導入

### 設計仕様

| 項目 | 決定 |
|------|------|
| チェックポイントの粒度 | 要件単位（条文単位ではない） |
| パターン記述方式 | ハイブリッド（自然言語パターン＋キーワード） |
| 法令横断の共通化 | しない（法令ごとに独立、重複OK） |
| severity | 固定デフォルト＋LLMが下方修正可（上方修正は不可） |
| keywordsの使い方 | プロンプトに含めてLLMが参照 |
| チェックポイント数 | 1法令あたり10前後 |

### チェックポイントのJSONスキーマ

```json
{
  "law_id": "string",
  "law_name": "string",
  "applicable_contract_types": ["string"],
  "checkpoints": [
    {
      "id": "string (例: HAISO-01)",
      "requirement": "string (要件名)",
      "legal_basis": ["string (条文参照、配列)"],
      "default_severity": "HIGH | MEDIUM | LOW",
      "description": "string (この要件の概要説明)",
      "compliant_patterns": ["string (自然言語で適合パターンを記述)"],
      "compliant_keywords": ["string (適合を示唆するキーワード)"],
      "violation_patterns": ["string (自然言語で違反パターンを記述)"],
      "violation_keywords": ["string (違反を示唆するキーワード)"]
    }
  ]
}
```

### articles_blockの生成方法

チェックポイントJSONとRAG取得条文から、LLMに渡す`{articles_block}`テキストを以下の形式で生成する。

**アーキテクチャ概要:**
```
【通常モード（checkpoint）】
LLMに渡す情報（articles_block + contract_text）:
┌─────────────────────────────────────────────────┐
│ ① チェックポイント一覧（事前定義・検証済みJSON）  │ ← 判定のアンカー（ブレ防止）
│    適合パターン + 違反パターン + keywords           │
├─────────────────────────────────────────────────┤
│ ② 法令条文テキスト（RAG取得、最大8件×600文字）    │ ← アンカーの補正・補完
├─────────────────────────────────────────────────┤
│ ③ 契約書テキスト                                  │ ← 判定対象
└─────────────────────────────────────────────────┘

【フォールバックモード（fallback）】※チェックポイント未設定時
LLMに渡す情報:
┌─────────────────────────────────────────────────┐
│ ① 法令条文テキスト（RAG取得、最大8件×600文字）    │ ← LLMがここから要件を導出（CoT）
├─────────────────────────────────────────────────┤
│ ② 契約書テキスト                                  │ ← 判定対象
└─────────────────────────────────────────────────┘
※ アンカーがないためブレやすい。結果に"mode":"fallback"フラグを付加。
```

```python
def build_articles_block(law_config: dict, articles_text: str) -> str:
    """
    法令条文テキストとチェックポイント定義から、
    LLMに渡すarticles_blockを生成する。
    
    Args:
        law_config: 検証済みチェックポイントJSON（1法令分）
        articles_text: RAGから取得した法令条文テキスト（最大8件×600文字を結合済み）
    """
    lines = []
    
    # セクション1: 法令条文テキスト（RAG取得）
    lines.append(f"【参照する法令条文（{law_config['law_name']} の要件根拠）】")
    if articles_text and articles_text.strip():
        lines.append(articles_text)
    else:
        lines.append("（有効な条文テキストが取得できませんでした）")
    lines.append("")
    
    # セクション2: チェックポイント一覧（事前定義）
    lines.append(f"【{law_config['law_name']} のチェックポイント一覧】")
    lines.append(f"（検証済み・全{len(law_config['checkpoints'])}項目）")
    lines.append("")
    
    for cp in law_config["checkpoints"]:
        lines.append(f"--- {cp['id']}: {cp['requirement']} ---")
        lines.append(f"根拠条文: {', '.join(cp['legal_basis'])}")
        lines.append(f"デフォルト重大度: {cp['default_severity']}")
        lines.append(f"概要: {cp['description']}")
        lines.append("")
        
        lines.append("✅ 適合パターン:")
        for p in cp["compliant_patterns"]:
            lines.append(f"  - {p}")
        if cp["compliant_keywords"]:
            lines.append(f"  適合キーワード例: {', '.join(cp['compliant_keywords'])}")
        lines.append("")
        
        lines.append("🔴 違反パターン:")
        for p in cp["violation_patterns"]:
            lines.append(f"  - {p}")
        if cp["violation_keywords"]:
            lines.append(f"  違反キーワード例: {', '.join(cp['violation_keywords'])}")
        lines.append("")
    
    return "\n".join(lines)
```

### プロンプトに追加するseverity判定ルール

現在のプロンプトの `【判定基準】` セクションを以下に置き換える。

```
【severity判定ルール】
各チェックポイントにはdefault_severityが定義されています。

● チェックポイントに基づく判定:
- 違反パターンに完全に該当する → default_severityをそのまま適用
- 部分的に該当する（一部は充足している）→ 1段階下げてよい（HIGH→MEDIUM、MEDIUM→LOW）
- 上方修正は不可（default_severityがMEDIUMのものをHIGHにしてはならない）
- 適合パターンに合致し違反テキストを引用できない → そのチェックポイントはCOMPLIANT

● 条文テキストによる補正での追加検出:
- チェックポイントに含まれない違反を条文テキストから検出した場合 → 上限MEDIUM
  （チェックポイント外の検出はHIGHにしない。重大な違反であればチェックポイントに追加すべき）
- gapsに「チェックポイント外の検出: [条文参照]に基づき検出」と記載すること

● チェックポイントと条文テキストが矛盾する場合:
- 条文テキストを優先する
- gapsに「チェックポイント[ID]と条文テキストの間に不整合あり: [具体的内容]」と記載すること

【判定基準】
- **COMPLIANT**: 全チェックポイントが適合、または別表委任・対象法令に無関係な契約
- **LOW**: 軽微な不明確点あり（問題テキストを引用できる）
- **MEDIUM**: 法令要件の不備あり（問題テキストを引用できる）
- **HIGH**: 重大な法令違反あり（問題テキストを契約書から明確に引用できる）
- **UNREADABLE**: 契約書テキストが断片的で判定不能

HIGH または MEDIUM と判定するには、問題となる契約書の記述を必ず直接引用してください。
引用できない場合は COMPLIANT としてください。
```

## P1-C: チェックポイントJSONの信頼性検証

### 課題

チェックポイントJSONはLLMが法令知識に基づいて生成したものであり、以下の誤りが含まれるリスクがある。

| リスク | 具体例 |
|--------|--------|
| 条文番号の誤引用 | `legal_basis`に記載した条文番号が実際には別の内容を規定している |
| 要件の過剰/不足 | 実際の条文にない要件を追加、または重要な要件を見落としている |
| 適合パターンの誤り | 実務上は適合とされない条件を「適合」と定義している |
| violation_keywordsの偏り | サンプル契約書の表現に引きずられ、実際の違反契約で使われる表現を網羅できていない |
| 会計基準の曖昧さ | 新リース会計基準は法令ではなく会計基準であり、violation/compliantの境界が不明確 |

さらに、サンプル契約書の作成もチェックポイントの作成も同じLLMが行ったため、**自分で作った違反パターンを自分で検出する循環構造**になっている。実際の契約書では全く別の表現で違反が記述される可能性がある。

### 信頼性レベルの選択

```
レベル1: チェックポイントJSONをそのまま信じる
  → LLM生成物を無検証で使うリスク（不採用）

レベル2: チェックポイントJSONを、RAG登録済み条文で検証してから使う ★採用
  → 一度だけ検証工程を挟む（デプロイ前）

レベル3: チェックポイントJSONは使わず、毎回条文から導出する
  → 毎回揺れるリスク（不採用）
```

### デプロイ前検証フロー

チェックポイントJSONを本番投入する前に、以下の検証を**一度だけ**実行する。

```python
def verify_checkpoints_against_law_text(law_config: dict, rag_search_fn) -> dict:
    """
    チェックポイントJSONの各項目について、
    RAG登録済みの法令条文と照合して検証する。
    
    Args:
        law_config: チェックポイントJSON（1法令分）
        rag_search_fn: RAG検索関数（クエリ→条文チャンクのリスト）
    
    Returns:
        検証結果レポート
    """
    report = {
        "law_id": law_config["law_id"],
        "law_name": law_config["law_name"],
        "verified_at": None,  # 検証日時
        "results": []
    }
    
    for cp in law_config["checkpoints"]:
        result = {
            "checkpoint_id": cp["id"],
            "requirement": cp["requirement"],
            "legal_basis": cp["legal_basis"],
            "verification": {}
        }
        
        # Step 1: legal_basisの各条文番号でRAGを検索
        found_articles = []
        missing_articles = []
        for article_ref in cp["legal_basis"]:
            # 条文番号でRAG検索
            chunks = rag_search_fn(article_ref)
            if chunks:
                found_articles.append({
                    "reference": article_ref,
                    "chunk_text": chunks[0]["text"][:500]  # 先頭500文字
                })
            else:
                missing_articles.append(article_ref)
        
        result["verification"]["found_articles"] = found_articles
        result["verification"]["missing_articles"] = missing_articles
        
        # Step 2: 条文テキストとdescriptionの整合性チェック（LLMで実行）
        if found_articles:
            articles_text = "\n".join([a["chunk_text"] for a in found_articles])
            # このプロンプトでLLMに検証させる
            verification_prompt = f"""
以下のチェックポイント定義が、法令条文と整合しているか検証してください。

【チェックポイント】
ID: {cp['id']}
要件: {cp['requirement']}
根拠条文: {', '.join(cp['legal_basis'])}
概要: {cp['description']}

適合パターン:
{chr(10).join('- ' + p for p in cp['compliant_patterns'])}

違反パターン:
{chr(10).join('- ' + p for p in cp['violation_patterns'])}

【RAGから取得した実際の条文テキスト】
{articles_text}

【検証項目】
1. legal_basisの条文番号は、実際の条文テキストと一致しているか？
2. descriptionの内容は、条文の規定内容と整合しているか？
3. compliant_patternsは、条文の要件を正しく反映しているか？
4. violation_patternsは、条文に照らして妥当な違反パターンか？
5. 条文に規定されているが、チェックポイントに含まれていない重要な要件はあるか？

JSON形式で回答してください:
{{
  "is_consistent": true/false,
  "issues": ["不整合がある場合の説明"],
  "missing_requirements": ["条文にあるがチェックポイントに含まれていない要件"],
  "suggested_corrections": ["修正提案"]
}}
"""
            result["verification"]["prompt_for_llm"] = verification_prompt
        
        # Step 3: missing_articlesがある場合は警告
        if missing_articles:
            result["verification"]["warning"] = (
                f"以下の条文がRAGで見つかりません: {', '.join(missing_articles)}。"
                f"法令テキストが未登録か、条文番号の記載が誤っている可能性があります。"
            )
        
        report["results"].append(result)
    
    return report
```

### 検証で問題が見つかった場合のフロー

```
1. verify_checkpoints_against_law_text() を実行
2. 各チェックポイントの検証結果を確認
   ├─ is_consistent: true → そのまま本番投入
   ├─ is_consistent: false → issues/suggested_correctionsに基づきJSONを修正
   └─ missing_articles あり → 条文番号を修正、またはRAGに法令テキストを追加登録
3. 修正後、再度検証を実行
4. 全チェックポイントがis_consistent: trueになったら本番投入
```

### 運用時のセーフティネット

チェックポイントJSONは事前検証済みであっても、以下の理由で判定が不正確になる可能性が残る。

- 法令改正によりチェックポイントが陳腐化
- 実際の契約書がチェックポイントの想定外の表現を使用
- チェックポイントに含まれない法令要件が存在

これに対処するため、**プロンプトにセーフティネット条項を追加する**（後述のプロンプト全体像を参照）。

```
【チェックポイントと条文テキストの関係】
チェックポイント一覧は判定の指針であり、判定のブレを抑えるためのアンカーです。
ただし、以下のルールに従ってください:

1. チェックポイントに記載がなくても、法令条文テキストに照らして
   明らかな違反がある場合は検出してください。
2. チェックポイントが指摘する要件が、実際の条文テキストと矛盾する場合は、
   条文テキストを優先してください。
3. チェックポイントに基づく判定と、条文テキストに基づく判定が
   異なる場合は、その旨をgapsに記載してください。
```

これにより、チェックポイントは**判定安定性のためのアンカー**として機能し、RAG取得した条文テキストが**チェックポイントの誤りを補正するフォールバック**として機能する。

### 定期レビュー体制（Phase 3）

本番運用安定後に、以下のプロセスを確立する。

| タイミング | アクション | 担当 |
|---|---|---|
| 法令改正時 | 該当法令のチェックポイントJSONを更新＋再検証 | エンジニア＋法務 |
| 四半期ごと | 誤判定ログを分析し、パターン/キーワードを追加修正 | エンジニア |
| 年次 | 法務チームによるチェックポイント全体レビュー | 法務 |

## P2: 別表の存在確認ロジック

### 課題

プロンプトで「別表のとおり」を充足と見なすルールがあるが、別表が実際に添付されていない場合に偽陰性（見逃し）が発生する。

### 実装指示

LLM呼び出し前の前処理として、以下のチェックを追加する。

```python
def check_attachment_references(contract_text: str, total_pages: int) -> list[str]:
    """
    契約書本文中の別表・別紙参照を検出し、
    実際に添付されている可能性を判定する。
    """
    import re
    
    warnings = []
    
    # 別表・別紙への参照を検出
    references = re.findall(r'(別表[１-９1-9]?|別紙[１-９1-9]?|附表[１-９1-9]?)', contract_text)
    unique_refs = set(references)
    
    if unique_refs:
        # 単ページの契約書で別表参照がある場合は警告
        if total_pages <= 1:
            warnings.append(
                f"契約書本文に{', '.join(unique_refs)}への参照がありますが、"
                f"文書が{total_pages}ページのみです。別表が添付されていない可能性があります。"
            )
        
        # 別表の内容（テーブルデータ等）がcontract_textに含まれているかチェック
        for ref in unique_refs:
            # 別表の見出しがテキスト内にあるか
            header_pattern = rf'{ref}\s*[（(].+[）)]'
            if not re.search(header_pattern, contract_text):
                warnings.append(
                    f"{ref}への参照がありますが、{ref}の内容がテキスト内に見つかりません。"
                    f"PDF解析で別表が読み取れていない可能性があります。"
                )
    
    return warnings
```

警告がある場合は、LLMに渡すプロンプトに以下を付加する。

```
【別表に関する注意】
以下の警告があります。「別表のとおり」で充足とみなす際は慎重に判断してください。
{warnings}
```

```python
prompt = f"""あなたはコンプライアンス専門家です。

- 意味不明な文字列のみの文書

- 契約書テキストが断片的で条文構造を認識できない

日本の契約書では詳細を別表・別紙に委任する形式が一般的な実務です。
{attachment_warnings}

{articles_block}

【チェックポイントと条文テキストの関係（重要）】
上記にはチェックポイント一覧と法令条文テキストの2種類の情報があります。
それぞれの役割と優先関係は以下の通りです:

● チェックポイント一覧 = 判定のアンカー（ブレ防止）
  - 適合パターン・違反パターン・キーワードは判定の基本指針として使用してください
  - これにより、同じ契約書に対する判定が毎回大きく変動することを防ぎます

● 法令条文テキスト = チェックポイントの補正・補完
  - チェックポイントに記載がなくても、条文テキストに照らして明らかな違反がある場合は検出してください
  - チェックポイントの記載が条文テキストと矛盾する場合は、条文テキストを優先してください
  - チェックポイントに基づく判定と条文に基づく判定が異なる場合は、その旨をgapsに記載してください

● 条文テキストが「（有効な条文テキストが取得できませんでした）」の場合
  - チェックポイント一覧のみで判定してください（条文補正なし）

---

【評価対象の契約書テキスト（文書順）】
{contract_text}

---

【評価手順】

**Step 1: チェックポイントごとに照合する**
上記のチェックポイント一覧を1つずつ確認し、契約書テキストと照合してください。
各チェックポイントの「適合パターン」「適合キーワード」「違反パターン」「違反キーワード」を参照してください。

**Step 2: 条文テキストによる補正チェック**
Step 1の結果を、法令条文テキストに照らして検証してください。
- チェックポイントがCOMPLIANTと判定した項目でも、条文に照らして問題がある場合は検出してください
- チェックポイントが違反と判定した項目でも、条文の趣旨からは問題ない場合は緩和してください

**Step 3: 適合/違反を判定する**
- 適合パターンに合致し、違反テキストを引用できない → COMPLIANT
- 「別表のとおり」「別紙参照」で委任 → COMPLIANT
- 違反パターンに完全に該当し、問題テキストを契約書から直接引用できる → default_severityを適用
- 違反パターンに部分的に該当する（一部は充足） → default_severityから1段階下げてよい
- 条文テキストにより追加検出した違反 → MEDIUM（チェックポイント外の検出であるため上限MEDIUM）
- default_severityの上方修正は不可（MEDIUMをHIGHにしてはならない）
- 当該法令に無関係な契約である → COMPLIANT

**Step 4: 総合判定**
全チェックポイントの中で最も高いseverityを、契約書全体のリスクレベルとする。

【判定基準】
- **COMPLIANT**: 全チェックポイントが適合、または対象法令に無関係な契約
- **LOW**: 軽微な不明確点あり（問題テキストを引用できる）
- **MEDIUM**: 法令要件の不備あり（問題テキストを引用できる）
- **HIGH**: 重大な法令違反あり（問題テキストを契約書から明確に引用できる）
- **UNREADABLE**: 契約書テキストが断片的で判定不能

HIGH または MEDIUM と判定するには、問題となる契約書の記述を必ず直接引用してください。
引用できない場合は COMPLIANT としてください。

COMPLIANT の場合は gaps を空リスト [] にしてください。
UNREADABLE の場合は gaps にテキスト品質の問題点を記載してください。"""
```

### フォールバックモード（mode="fallback"）のプロンプト

チェックポイントJSONが未設定（空）の法令に対して使用する。
条文テキストのみを頼りにCoTで判定する。判定の再現性は通常モードより低い。

```python
fallback_prompt = f"""あなたはコンプライアンス専門家です。
以下の法令条文を読み、契約書がその要件を満たしているかを評価してください。

  判定の再現性が通常より低い可能性があります。

【評価対象外の文書】

- 領収書・請求書・納品書・名刺
- 意味不明な文字列のみの文書

その要件が別表・別紙に記載されているとみなして COMPLIANT としてください。
日本の契約書では詳細を別表・別紙に委任する形式が一般的な実務です。
{attachment_warnings}

- HIGH と判定するには重大な法令違反が明確に引用できる必要があります

【判定基準】
- **COMPLIANT**: 法令要件を適切に満たしている、または対象法令に無関係な契約
- **LOW**: 軽微な不明確点あり（問題テキストを引用できる）
- **MEDIUM**: 法令要件の不備あり（問題テキストを引用できる）
- **HIGH**: 重大な法令違反あり（問題テキストを契約書から明確に引用できる）

HIGH または MEDIUM と判定するには、問題となる契約書の記述を必ず直接引用してください。
引用できない場合は COMPLIANT としてください。

COMPLIANT の場合は gaps を空リスト [] にしてください。
UNREADABLE の場合は gaps にテキスト品質の問題点を記載してください。"""
```

### 2つのモードの比較

| 観点 | 通常モード（checkpoint） | フォールバックモード（fallback） |
|------|--------------------------|----------------------------------|
| 使用条件 | チェックポイントJSONが設定済み | チェックポイントJSONが空 |
| 判定のアンカー | チェックポイント一覧 | なし（条文テキストのみ） |
| 要件の導出 | 事前定義済み | LLMが条文から毎回導出（CoT） |
| 判定の再現性 | 高い | 低い（同一入力でもブレうる） |
| severity上限 | HIGH（チェックポイント由来）/ MEDIUM（条文補正由来） | HIGH（制限なし） |
| UI表示 | 通常表示 | 「⚠ チェックポイント未設定」警告を付加 |
| 推奨用途 | 本番判定 | 新規法令の試験的判定、チェックポイント作成前の事前評価 |

```python
        }
    ]
}
```

## テスト計画

上記改修を実装後、以下の2段階でテストを実施すること。

### Phase 1: チェックポイントJSON検証（P1-Cの実行）

デプロイ前に`verify_checkpoints_against_law_text()`を実行し、全チェックポイントの整合性を確認する。

#### 検証テスト1-1: 廃掃法チェックポイント（HAISO-01〜10）
- **実行**: 各チェックポイントの`legal_basis`でRAG検索し、条文テキストを取得
- **確認項目**:
  - 10件すべてで条文テキストがヒットするか（missing_articlesが0か）
  - 特に「施行令第6条の2第4号イ〜ヘ」が正しくチャンキングされてRAG検索可能か
  - descriptionの内容が実際の条文と整合しているか
- **不合格基準**: missing_articlesが2件以上、またはis_consistent: falseが3件以上

#### 検証テスト1-2: 取適法チェックポイント（TORI-01〜10）
- **実行**: 同上
- **確認項目**:
  - 取適法第3条（書面等による明示義務）、第4条（60日ルール）、第5条（禁止行為）の条文が取得できるか
  - 第12条・第13条（中途解除の予告義務）が正しく検索されるか
- **不合格基準**: 同上

#### 検証テスト1-3: 新リース会計基準チェックポイント（LEASE-01〜09）
- **実行**: 同上
- **特記事項**: 新リース会計基準は会計基準であり法令ではないため、RAGに登録されていない可能性がある。未登録の場合は、企業会計基準第34号のテキストをRAGに追加登録すること。
- **確認項目**:
  - 第17項（リース期間）、第20項（使用権資産の取得原価）、第22項（リース負債の測定）等の条文が取得できるか
- **不合格基準**: 同上

#### 検証テスト1-4: チェックポイント未カバー領域の洗い出し
- **実行**: 各法令について、RAG登録済み条文の上位20チャンクを取得し、チェックポイントでカバーされていない重要要件がないか確認

### 判定品質のモニタリング指標

本番運用開始後、以下の指標を継続的にモニタリングする。

| 指標 | 計算方法 | 目標値 |
|------|----------|--------|
| 再現率（Recall） | 既知の違反契約のうち正しくHIGH/MEDIUMと判定された割合 | ≥90% |
| 適合率（Precision） | HIGH/MEDIUM判定のうち実際に違反があった割合 | ≥80% |
| 再現性（Consistency） | 同一入力5回実行でリスクレベルが一致する割合 | ≥90%（通常）/ ≥70%（fallback） |
| UNREADABLE率 | 全判定中のUNREADABLEの割合 | ≤5%（高すぎる場合はOCR改善が必要） |
| セーフティネット発動率 | 条文テキスト補正で判定が変わった割合 | 参考値として記録（高すぎる場合はチェックポイントの品質を見直し） |
| フォールバック使用率 | 全判定中のfallbackモードの割合 | 参考値として記録（高すぎる場合はチェックポイント作成を優先） |
