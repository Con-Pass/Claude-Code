## 1. プロダクトビジョンと最終到達点

### 1.1 ビジョンステートメント

> **ConPassは「商取引インテリジェンス・プラットフォーム」として、契約管理の効率化を超え、商取引における情報格差を解消し、企業規模に関係なく健全な取引ができる社会を実現する。**

ConPassが目指すのは単なる契約管理SaaSではない。契約 → 取引 → 決済のトレーサビリティを一気通貫で提供し、日本の商取引における「信用インフラ」となることである。

現在の契約管理市場では、大企業は法務部門と顧問弁護士を擁し、有利な契約条件を勝ち取る一方、中小企業は情報・ノウハウの不足から不利な条件を受け入れざるを得ないという構造的な非対称性が存在する。ConPassはAIとデータの力でこの格差を埋め、すべての企業が対等な立場で商取引に臨める世界を創出する。

### 1.2 顧客への約束（v2追加）

> **2030年、日本の中小企業の経営者は、ConPassを開くだけで、すべての契約の状況を一目で把握できる。更新が必要な契約、リスクのある取引先、業界標準より不利な条件の契約。それらがAIによって自動的にハイライトされ、何をすべきかが明確に示される。専門の法務部門がなくても、大企業と対等に取引できる。それがConPassの約束である。**

### 1.3 社会に届ける3つの価値

#### (1) 取引の非対称性解消

| 現状の課題 | ConPassが提供する解決策 |
|---|---|
| 大企業のみが業界標準の契約条件を把握 | ベンチマークDBにより中小企業も業界標準を参照可能 |
| 法務リソースの有無が交渉力を左右 | AIリスクチェック・交渉戦略AIが法務機能を民主化 |
| 過去の交渉履歴が属人的知識に依存 | ナレッジグラフが交渉履歴を組織知として蓄積・活用 |

ベンチマークDBは匿名化された契約条件の業界横断統計であり、中小企業が「業界ではこの条件が標準です」と根拠を持って交渉できるようになる。これは情報の非対称性を根本から覆す仕組みである。

#### (2) 商取引の健全性可視化

| 現状の課題 | ConPassが提供する解決策 |
|---|---|
| 契約と実際の取引実態の乖離が不可視 | 契約 → 証憑 → 決済の3点照合による自動検証 |
| 取引先の信用リスクが事後的にしか判明しない | 取引先信用情報連携（TDB/TSR）によるリアルタイム監視 |
| コンプライアンス違反の発見が遅延 | 定期クローリング＋プッシュ型アラートによる予防的監視 |

契約書を「結んで終わり」にしない。契約の履行状況を継続的にモニタリングし、取引の健全性をリアルタイムに可視化することで、不正や不履行を未然に防ぐ。

#### (3) 経営判断の民主化

| 現状の課題 | ConPassが提供する解決策 |
|---|---|
| 経営ダッシュボードは高額なBIツールが前提 | ConPassが契約データから経営指標を自動生成 |
| 業界動向の把握に専門コンサルが必要 | クロス業界ベンチマークによるセルフサービス分析 |
| 予測的リスク分析は大企業の専売特許 | AI予測モデルが中小企業にも予測的リスク分析を提供 |

契約データは経営の最上流に位置する情報である。ConPassはこのデータを構造化・分析し、専門家不在でも精度の高い経営判断ができる環境を提供する。

### 1.4 2Moat集中戦略（v2改訂：旧5Moat戦略から集中へ）

v1では5つのMoatの同時構築を計画していたが、レビューを踏まえ**2つの中核Moatへの集中投資**に転換する。バフェットの「穴を10個浅く掘るより、1つの穴を深く掘れ」の原則に従い、限られたリソースを最も模倣困難で価値の高いMoatに集中する。

│   ┌───────────────────────────────────────────────────┐     │
│   │         中核Moat 1: BPO統合                        │     │
│   │         開発投資の50%以上を集中                     │     │
│   │                                                   │     │
│   │  ・日本パープル7,000社・20,000事業所のBPO基盤       │     │
│   │  ・紙→電子のブリッジ（物理的参入障壁）             │     │
│   │  ・スキャン→OCR→AI解析→保管→廃棄の一気通貫       │     │
│   │  ・スイッチングコスト：物理的依存による高い障壁     │     │

│                           │                                 │
│                           ▼                                 │
│   ┌───────────────────────────────────────────────────┐     │
│   │         中核Moat 2: ベンチマークDB                  │     │

│   │  ・匿名化契約条件の業界横断統計                     │     │
│   │  ・利用企業増加 → データ価値向上 → 正のFBループ    │     │
│   │  ・55期中にBeta版を建設・不動産業界で限定公開       │     │
│   └───────────────────────────────────────────────────┘     │
│                                                             │
│   ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─     │
│   付随的成果（独立Moatとしては扱わない）                     │
│                                                             │
│   ・ナレッジグラフ ← BPOデータ蓄積の副産物                  │
│   ・規制準拠認証   ← エンタープライズ拡大時に段階取得        │
│   ・業界特化ナレッジ ← バーティカル展開に伴い自然蓄積       │
└─────────────────────────────────────────────────────────────┘
```

#### 中核Moat 1：BPO統合の深掘り

| 項目 | 定量化 |
|---|---|
| **物理的参入障壁** | BPO網の新規構築コスト：推定10億円以上、構築期間：最低3年 |
| **模倣コスト** | 倉庫インフラ＋配送網＋スキャニング設備＋機密廃棄体制の総合投資 |
| **スイッチングコスト** | 紙原本の物理移管＋再スキャン＋データ移行で推定3-6ヶ月・数百万円 |
| **開発投資配分** | 全開発投資の50%以上をBPO統合の深化に集中 |
| **対象顧客のチャーン低減効果** | BPO統合顧客のチャーンレートは非BPO顧客の60-70%と仮説（要検証） |

**BPO統合の耐久性に関する留意事項：**

電子契約の普及に伴い紙契約の比率は低下していくが、以下の理由からBPO統合Moatは中長期的に有効である。

1. **過去契約の電子化需要**：既存の紙契約書（数十年分）のデジタル化ニーズは今後10年以上継続
2. **混在管理の必要性**：紙と電子の混在期間は日本の中小企業では最低でも10年以上
3. **進化するBPO価値**：紙→電子のブリッジから、データ品質管理・コンプライアンス監査支援へ価値がシフト

#### 中核Moat 2：ベンチマークDB（前倒し戦略）

| 項目 | v1計画 | v2改訂 |
|---|---|---|
| **公開時期** | MS3（2027.03） | 55期中にBeta版限定公開 |
| **最小データ要件** | 50社・500契約 | 45社の既存データから抽出可能な指標で開始 |
| **初期対象業界** | 全業界 | 建設・不動産業界に限定 |
| **データ入口** | 匿名化パイプライン完成後 | 既存データの手動抽出＋段階的自動化 |

**45社の既存データから抽出可能なベンチマーク指標：**

| 指標カテゴリ | 具体的な指標 | 必要データ量 |
|---|---|---|
| 支払条件 | 支払サイト分布（月末締め翌月末 / 翌々月末等） | 30社以上 |
| 契約期間 | 平均契約期間、自動更新率 | 20社以上 |
| 損害賠償 | 損害賠償上限額の分布パターン | 20社以上 |
| 解約条件 | 解約通知期間の業界平均 | 20社以上 |
| 秘密保持 | 秘密保持期間の傾向 | 15社以上 |

#### Moat間のシナジー（v2改訂）

BPO統合（Moat 1）とベンチマークDB（Moat 2）は相互に強化し合う関係にある。

```
BPO統合 ──→ 紙契約のデジタル化 ──→ データ量の増加 ──→ ベンチマークDBの精度向上
    ▲                                                            │
    │                                                            │
    └────── ベンチマーク参照顧客の増加 ← 新規顧客獲得 ──────────┘
```

ナレッジグラフ、規制準拠認証、業界特化ナレッジは、この2つの中核Moatを強化する「付随的成果」として、事業展開の過程で自然に蓄積される。独立したMoatとしてリソースを分散させることはしない。

**技術的強み：**
- GPT-4oを活用したAI-OCR台帳自動生成は98%の精度を実現

- TinyMCEエディタとWord実務の乖離（ユーザー体験の低下）

- 電子契約連携（GMO Sign）の断絶
- CTO不在による技術的リーダーシップの欠如（55期中に採用予定）

### 2.3 課題と制約 - ST比率10%の原因分析

現在のスタンダード（ST）プラン比率は約10%にとどまっている。この低さは売上成長とARPA向上の最大のボトルネックである。

```
ST比率10%の原因構造:

┌──────────────────────────────────────────────────────────┐
│                ST比率 10% （目標 70%+）                    │
├──────────────┬──────────────┬─────────────┬──────────────┤
│  機能差別化   │  体験品質    │  営業構造   │  価格認知    │
│  の不足      │  の課題      │  の課題     │  の課題      │
├──────────────┼──────────────┼─────────────┼──────────────┤
│・AIアシスタント│・Word乖離の │・ライトから │・ST月額5万の │
│ がプル型のみ  │  エディタ   │ のアップ   │  価値が不明確│
│・WF条件分岐  │・電子契約   │  セル導線  │・AIオプション│
│  不可        │  連携断絶   │  が弱い    │  の価値訴求  │
│・RAG知識基盤 │             │            │  不足       │
│  が貧弱      │             │            │             │
└──────────────┴──────────────┴─────────────┴──────────────┘
```

**根本原因の分析：**

1. **機能差別化の不足：** STプランでしか使えない「圧倒的な価値」がない。AIアシスタントはプル型のみで、ユーザーが質問しなければ価値を体感できない。
2. **体験品質の課題：** TinyMCEエディタはWord実務との乖離が大きく、ユーザーが「使いにくい」と感じる最大の要因。
3. **営業構造の課題：** ライトプランからSTプランへのアップセル導線が体系化されていない。
4. **価格認知の課題：** STプラン月額50,000円の費用対効果が定量的に示されておらず、AIオプション15,000円/月の価値訴求も不足。

## 6. AI・技術戦略

### 6.1 RAGアーキテクチャの進化（9ナレッジソース計画）

ConPassのAI価値の中核は、9つのナレッジソースを統合したRAG（Retrieval-Augmented Generation）アーキテクチャにある。

**現在のアーキテクチャ：**

```
┌─────────────────────────────────────────────────────────┐
│                   クエリ処理                              │
│        ユーザー質問 → クエリ分析 → 検索意図推定            │
├─────────────────────────────────────────────────────────┤
│                  Hybrid Search                           │
│          dense (embedding) + sparse + RRF                │

## 11. プッシュ型AIへの転換

### 11.1 プル型からプッシュ型への転換

```
【55期まで：プル型AI】              【56期以降：プッシュ型AI】

ユーザー → 質問 → AI → 回答        AI → 検知 → アラート → ユーザー

・ユーザーが質問しないと          ・契約更新期限の自動通知
  価値を発揮しない               ・法改正による影響契約の特定
・AIの存在を忘れられる            ・取引先信用変動の即時警告
                                  ・コンプライアンス違反の予防検知
                                  ・支払条件の業界標準との乖離通知
```

この転換により、AIの価値がユーザーの能動的行動に依存しなくなり、ST/AIオプションの価値訴求が格段に容易になる。「価値の階段」のStep 2（「守れる」）の中核をなす機能である。

### 11.2 プッシュ型AIが生む定量的価値

| アラート種別 | 推定防止損失額/年/社 | 顧客への価値表示例 |
|---|---|---|
| 契約更新漏れ検知 | 300万円 | 「今月3件の契約更新期限が近づいています」 |
| 法改正影響通知 | 100万円 | 「建設業法改正により、5件の契約条項見直しが必要です」 |
| 取引先信用変動 | 500万円 | 「取引先Aの信用スコアが低下。未回収リスク○万円」 |
| 不利条件アラート | 200万円 | 「御社の支払条件は業界平均より15日長い（不利）」 |

## 12. リスクと対策

| リスク | 影響度 | 発生確率 | 対策 |
|---|---|---|---|
| GPT-4o等のLLM API料金高騰 | 高 | 中 | マルチLLM戦略（OSS LLMへの段階的移行検討）、キャッシュ最適化 |
| 競合（LegalForce等）の急速な機能追加 | 高 | 高 | 2Moat（BPO統合＋ベンチマークDB）の早期確立に集中 |
| クラウドサインが契約管理機能を無料提供 | 高 | 中 | BPO統合という物理的Moatは価格競争の影響を受けにくい |
| ベンチマークDBのデータ量不足 | 高 | 中 | 55期からMVFで早期開始、BPO経由のデータ取込みで量を確保 |
| CTO採用の遅延・失敗 | 高 | 中 | 採用チャネルの多様化、年収800-1,000万円の競争力ある提示 |
| 開発リソースの不足 | 高 | 高 | BrainStation-23との継続的協業、段階的な内製化 |
| 電子契約普及によるBPO価値減衰 | 中 | 中 | BPO価値のシフト（紙→電子ブリッジから、データ品質管理・監査支援へ） |
| 個人情報保護規制の強化 | 中 | 中 | 匿名化パイプラインの堅牢化、法務顧問との定期レビュー |
| 建設・不動産業界集中のリスク | 中 | 低 | 57期から第2バーティカルへの計画的展開 |
| BPO統合顧客のチャーン仮説の否定 | 中 | 低 | 55期中にBPO顧客 vs 非BPO顧客のチャーン差分を検証 |

## 主要な改訂ポイント（v1 → v2）

### 構造的変更

1. **5Moat → 2Moat集中戦略への転換**
   - v1: 5つのMoat（ベンチマークDB、ナレッジグラフ、規制準拠認証、BPO統合、業界特化ナレッジ）を同時構築
   - v2: BPO統合（開発投資50%以上）を中核Moat、ベンチマークDBを第2Moatに集中。残り3つは「付随的成果」として位置づけ
   - 根拠: バフェット「穴を10個浅く掘るより、1つの穴を深く掘れ」

2. **BPOフライホイールの明示的設計**
   - v1: BPO統合の価値は語られていたが、フライホイールとして設計されていなかった
   - v2: BPO → データ蓄積 → AI精度向上 → 顧客価値増大 → リテンション向上 → 口コミ → 新規顧客の循環構造を明示化し、各ステップにメトリクスを設定
   - 根拠: ベゾス「BPOフライホイールを明示的にデザインし、回転速度メトリクスを定義せよ」

3. **ベンチマークDB前倒し（MS3→55期中Beta版）**
   - v1: MS3（2027.03）で業界別ベンチマーク公開
   - v2: 55期中に既存45社のデータからBeta版を建設・不動産業界の限定顧客に公開
   - 根拠: 孫正義「データは時間の関数だ。早く始めた者が勝つ」、ゴーディン「今すぐ無料の業界別契約条件レポートを公開せよ」

4. **ロードマップの期ベース再整理**
   - v1: MS0-MS5の6段階マイルストーン
   - v2: 55期-59期の5期ベースに再整理。各期で「何をやり、何をやらないか」を明確化
   - 根拠: 計画の実行可能性向上と社内コミュニケーションの明確化

### メトリクス・数値の変更

5. **顧客視点メトリクスの導入**
   - v1: ARR、ARPA、ST比率等の供給者視点KPIのみ
   - v2: TTFV、WAU率、NPS、Feature Adoption Rate、Contract Search Timeを追加
   - 根拠: ベゾス「メトリクスを顧客視点で再定義せよ」

6. **共通数値前提の統一**
   - v1: 59期顧客数254社、ARR 1.27億円、ARPA全期50万円
   - v2: 59期顧客数500社、ARR 2.5億円、ARPA 55期50万→59期80万円
   - v1: 月次チャーンレート混在（0.5%/1.5%/2.0%）
   - v2: 基準1.5%→55期末1.3%→57期1.0%→59期0.8%に統一

7. **初期集中業界の明確化**
   - v1: 7業界同時展開（不動産、製造、卸売、士業、金融、IT、介護）
   - v2: 建設・不動産業界に55-57期リソース70%集中
   - 根拠: 孫正義「1つのセグメントでNo.1を取れ」、ゴーディン「Smallest Viable Audienceを見つけよ」

### 新規追加セクション

8. **「価値の階段」設計の追加**
   - Step 1: 台帳化（「見える」）← BPO統合
   - Step 2: 監視（「守れる」）← プッシュ型AI
   - Step 3: 洞察（「戦える」）← ベンチマークDB
   - 根拠: ベゾス「ARPA引き上げは『価格改定』ではなく『価値の階段』で設計せよ」

9. **顧客への約束（ビジョンの顧客視点表現）の追加**
   - 根拠: ベゾス「Working Backwards」、ゴーディン「スペックではなくストーリーを語れ」

10. **CTO採用の前倒し（56期→55期中）**
    - v1: 56期から正社員エンジニア採用開始（年収600万円）
    - v2: 55期中にCTOクラスを採用（年収800-1,000万円）
    - 根拠: バフェット・ベゾス「技術的リーダーシップを社内に置くべき」

11. **BPO統合パッケージ料金の設計**
    - v1: BPOサービスは「別会計」
    - v2: SaaS料金にBPOパッケージを統合（月額+15,000〜40,000円）
    - 根拠: 孫正義「BPOを統合したパッケージ料金を設計し、ARPAに転換せよ」

12. **「業界別契約条件レポート」月次無料配信**
    - v1: なし
    - v2: 55期中から毎月配信開始（Permission Marketing入口）
    - 根拠: ゴーディン「Permission Marketingの入口として、無条件で価値を届けよ」
