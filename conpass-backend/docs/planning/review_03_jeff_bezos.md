# ベゾスレビュー：ConPass 5ヵ年事業計画

> レビュアー：ジェフ・ベゾス（ペルソナ）
> レビュー日：2026年2月17日
> 対象：ConPass 5ヵ年事業計画書（市場調査・プロダクト戦略・財務モデル・GTM営業戦略・整合性検証レポート）

---

## 総合評価：B+

この計画は「よくできたMBA的な事業計画」である。分析は丁寧で、数字の積み上げも論理的だ。しかし、私がAmazonで最も重視してきたこと――**顧客から逆算してすべてを設計する**――という原則が、この計画の根幹に欠けている。

計画書のどこにも「顧客の声」がない。顧客が何に困っていて、夜眠れないほど悩んでいることは何なのか。顧客がConPassを使った瞬間に「これだ」と感じる体験は何なのか。それが書かれていない。代わりに、市場規模、競合分析、ARPA引き上げ戦略が並んでいる。これは**供給者の論理**で書かれた計画であり、**顧客の言葉**で書かれた計画ではない。

It's always Day 1だ。Day 2は停滞であり、それに続くのは苦痛を伴う衰退であり、そして死だ。この計画にはDay 2の兆候がいくつか見える。

## 1. 顧客中心設計の評価

### 致命的な問題：顧客のペルソナが存在しない

469ページ（5ファイル合計）に及ぶ事業計画書の中に、**たった一人の具体的な顧客の物語**も書かれていない。

Amazonでは、すべてのプロジェクトは「プレスリリース」から始まる。顧客がこの製品を手にしたとき、どんな体験をするのか。どんな問題が解決されるのか。それを最初に書く。ConPassの計画には、これがない。

**具体的に欠けている要素：**

- 従業員30名の建設会社の総務担当者・田中さんが、毎月どれだけの時間を契約書の管理に費やしているのか
- その田中さんがConPassを導入した翌日、何が変わるのか
- 田中さんが上司に「ConPassを導入してよかった」と報告するとき、どんな数字を見せるのか
- 田中さんがConPassを解約しようと思うのは、どんなときか

市場調査セクションでは「中小企業のSaaS導入率は10~15%にとどまる」と書いてある。では、残り85~90%の企業は**なぜ**導入していないのか？ 価格の問題なのか、認知の問題なのか、それとも「紙でも別に困っていない」と思っているのか。この問いに対する答えが、計画のどこにもない。

### 「BPO統合」は顧客視点か？

BPO統合をConPassの最大の差別化要因としている。だが、これは**供給者の都合**から生まれた差別化ではないか。

顧客は「BPO統合されたSaaS」を求めているのか？ それとも「紙の契約書を簡単に探せるようにしたい」だけなのか？ もし後者なら、スマートフォンで契約書を撮影するだけでAI-OCRが台帳化してくれる機能のほうが、BPOより圧倒的に顧客体験として優れている可能性がある。

BPO統合は日本パープルの既存アセットを活用するという**経営判断**としては合理的だ。だが、それが顧客にとっての最善かどうかは、検証されていない。

### Working Backwards が欠如している

この計画には「59期にARR 1.27億円」という財務目標はあるが、「59期に顧客がどんな体験をしているか」のビジョンが弱い。プロダクト戦略のビジョンステートメント「商取引インテリジェンス・プラットフォーム」は壮大だが、中小企業の総務担当者にとって「商取引インテリジェンス」は**意味不明**だ。

**私なら、こう書く：**

> 「2030年、日本の中小企業の経営者は、ConPassを開くだけで、すべての契約の状況を一目で把握できる。更新が必要な契約、リスクのある取引先、業界標準より不利な条件の契約。それらがAIによって自動的にハイライトされ、何をすべきかが明確に示される。専門の法務部門がなくても、大企業と対等に取引できる。それがConPassの約束である。」

## 2. フライホイールの評価

### フライホイールの設計は存在する――だが、回転を始める力が弱い

プロダクト戦略のMoat戦略とベンチマークDB構想は、フライホイールの設計としては評価できる。

```
顧客が増える
  → 契約データが蓄積される
    → ベンチマークDBの価値が高まる
      → 新規顧客がConPassを選ぶ理由が強まる
        → 顧客がさらに増える
```

Amazonのフライホイールと同じ構造だ。我々の場合は：

```
低価格 → 顧客体験の向上 → トラフィック増加 → 出品者増加 → 品揃え充実 → 顧客体験の向上...
```

しかし、ConPassのフライホイールには**決定的な弱点**がある。

**回転開始に必要なデータ量が多すぎる。**

ベンチマークDBの「限定公開」に必要な最低データ量は「50社・500契約」と記載されている。しかし、55期末の顧客数目標は79社だ。しかもすべての顧客がデータ提供に同意するわけではない。つまり、フライホイールが実際に回り始めるのは56期後半以降であり、それまでの1年半以上は**フライホイールなしで戦わなければならない**。

Amazonでは「フライホイールを最速で回し始める」ことに執拗にこだわった。ConPassの計画には、この**初期の回転力**をどう生み出すかの戦略が欠けている。

### 顧客をフライホイールに組み込めていない

もう一つの問題は、フライホイールの中に「顧客体験の向上」が明示的に組み込まれていないことだ。データが増えればDBの価値は上がる。だが、それが個々の顧客の日常業務をどう改善するかの設計が不十分だ。

Amazonのフライホイールでは、すべての要素が**顧客体験の向上**に帰結する。ConPassのフライホイールは、現状では「データの蓄積→プラットフォームの強化」という**供給者側の好循環**に見える。

### BPOフライホイールの可能性

一方で、BPO統合にはフライホイールの種がある。

```
BPOで紙契約を回収
  → ConPassにデータが蓄積
    → AIの精度が向上
      → 顧客の業務効率が改善
        → BPO＋ConPassの継続利用が定着
          → スイッチングコストが上昇
            → チャーンレート低下
```

この「BPOフライホイール」は計画で暗黙に想定されているが、**明示的に設計されていない**。これを意図的にデザインし、回転速度を測定するメトリクスを定義すべきだ。

## 3. 致命的な欠陥

### 欠陥1：2つのARR計画が共存し、どちらが本当の計画か不明

整合性検証レポートが正直に指摘している通り、施策別積上ARR（59期：2.85億円）と顧客数ベースARR（59期：1.27億円）の間に**2.2倍の乖離**がある。

これは重大な問題だ。営業チームは2.85億円を目指して動くのか、それとも1.27億円で良いのか。投資判断は2.85億円の前提で行うのか、1.27億円の前提で行うのか。

Amazonでは、数字は一つしかない。その数字にコミットし、その数字で評価される。2つの数字が共存する計画は、**どちらにもコミットしていない**のと同じだ。

**私の忠告：** 顧客数ベースARR 1.27億円を公式計画とし、施策別積上ARRは「楽観シナリオの内部参考値」として位置づけよ。そして、公式計画を達成するために何が必要かを、容赦なく逆算せよ。

### 欠陥2：ARPA 50万円の壁を突破する具体策が不明

5年間のベースシナリオARPAが498~501千円で**ほぼ横ばい**だ。一方で、ARPA 100万円への引き上げが「重要施策」として記載されている。この矛盾は致命的だ。

ARPA引き上げの道筋として「ST比率向上」「ユーザー課金導入」「オプション拡充」が挙げられているが、これらは「何をするか」のリストであって、「なぜ顧客がより多く支払うのか」の答えではない。

**顧客がより多く支払う理由は、たった一つだ。** それは、支払う金額以上の価値を受け取っていると実感するからだ。月額3万円のライトプランで「十分だ」と感じている顧客に、月額6万円のスタンダードプランを買わせるには、「これなしでは業務が回らない」レベルの価値が必要だ。

現状のST比率が10%（プロダクト戦略記載）というのは、**顧客が追加の2万円分の価値を感じていない**ことの証拠だ。この問題の根本原因分析は行われているが、解決策がプロダクト機能の追加に偏っている。顧客がなぜ「ライトで十分」と感じているのかのインタビューデータがなく、仮説に基づいた機能開発になっている。

### 欠陥3：254社という到達点のスケール感

59期（2030年）の目標が254社・ARR 1.27億円だ。SAM浸透率0.13%と自ら記載しており、これが「保守的」であることは理解している。

だが、問題は**保守的かどうかではなく、254社で何が実現できるか**だ。

- 254社では、ベンチマークDBのデータ量はネットワーク効果を発揮するには不十分
- 254社では、プラットフォームとしての価値は限定的
- 254社では、「商取引インテリジェンス・プラットフォーム」というビジョンは絵に描いた餅

ビジョンの壮大さと計画の保守性のギャップが大きすぎる。**ビジョンを縮小するか、計画を拡大するか、どちらかを選べ。**

### 欠陥4：チャーンレートの前提が3種類存在する

- 財務モデル（売上計画）：月次0.5~0.7%
- 財務モデル（LTV計算）：月次1.5%
- GTM営業戦略：月次2.0%（改善前）~1.2%（改善後）

SaaS事業において、チャーンレートは最も重要なメトリクスの一つだ。それが3つの異なる前提で計算されていることは、**この事業がチャーンの実態を把握していない**ことを意味する。

まず現実のチャーンレートを正確に測定せよ。コホート別に。月次で。プラン別に。解約理由別に。それができていない計画は、砂上の楼閣だ。

### 欠陥5：開発リソースの集中リスク

BrainStation-23（外部委託）への開発依存度が極めて高い。55期の業務委託費は月額90万円で、正社員エンジニアは0名。56期にようやく1名の正社員エンジニア採用を開始する。

5つのMoat、9つのRAGナレッジソース、マルチエージェントアーキテクチャ、ベンチマークDB、API公開...。これだけの技術的野心を、外部委託主体の開発体制で実現できるのか。

Two-pizza teamの原則は「小さなチームが自律的に動く」ことだ。外部委託チームは、製品のビジョンを深く理解し、顧客のフィードバックを直接受け取り、自律的に意思決定できるか？ 大いに疑問だ。

## 4. 評価できる点

### 4.1 BPO統合は本物のMoat

「模倣困難な競争優位」という言葉は濫用されがちだが、日本パープルの7,000社・20,000事業所のBPO基盤は**本物のMoat**だ。純粋なSaaS企業がこれを模倣するには、物理的なオペレーション網を一から構築する必要があり、数年と数十億円の投資が必要になる。

Amazonがフルフィルメントセンターで構築した物理的なMoatと同じ構造だ。テクノロジーだけでは模倣できない、オペレーションの厚みがある。これは正しい戦略だ。

### 4.2 中小企業×低価格のポジショニング

競合分析のポジショニングマップが示す通り、中小企業向け×低価格×導入容易のポジションはConPassの独壇場だ。上位の競合（ContractsONE、LegalForce等）がこのセグメントに降りてくるのは、彼らのビジネスモデルと矛盾する。これは「競合が来ない場所で戦う」というAmazonが初期に書籍から始めた戦略と共通する。

### 4.3 市場調査の質

市場調査セクションの質は高い。TAM/SAM/SOMの算出根拠が明確で、複数の調査機関のデータをクロスリファレンスしている。5層利用者モデルも独創的で、市場拡大の段階的なストーリーとして説得力がある。

### 4.4 フライホイールの種

ベンチマークDBの構想、ネットワーク効果の設計、テンプレートマーケットプレイスの構想は、いずれもプラットフォームビジネスのフライホイールとして正しい方向性だ。設計の骨格は良い。問題は実行のスピードと順序だ。

### 4.5 感度分析の誠実さ

財務モデルの感度分析とシナリオ分析は誠実に行われている。悲観シナリオで60期ARR 9,000万円・累計損益-9,164万円というリスクを正直に示している点は、投資家として信頼できる。

## 5. ベゾスからの助言

### 助言1：Day 1から始めよ――今すぐ10社の顧客に会いに行け

計画書を書く前に、あるいは書き直す前に、今すぐ10社の既存顧客と深いインタビューを行え。30分の定型アンケートではなく、2時間の対話だ。

聞くべきことは：

- ConPassを導入する前、契約書管理で最も困っていたことは何か
- ConPassを導入して、最も変わったことは何か
- ConPassで最も使っている機能は何か、逆に全く使っていない機能は何か
- もしConPassが明日なくなったら、最も困ることは何か
- ConPassに一つだけ機能を追加するなら、何がほしいか

この10社のインタビュー結果が、計画書のすべてのセクションに反映されるべきだ。

### 助言2：メトリクスを顧客視点で再定義せよ

現在のKPIは供給者視点だ。ARR、ARPA、ST比率、顧客数。これらは結果指標であり、顧客の幸福度を直接測定していない。

**以下のメトリクスを最優先で導入せよ：**

| メトリクス | 定義 | 目標 |
|-----------|------|------|
| **Time to First Value** | 契約締結から顧客が「便利だ」と実感するまでの日数 | 7日以内 |
| **Weekly Active Users / Total Users** | 週次アクティブ率 | 60%以上 |
| **Contract Search Time** | 契約書を探すのにかかる平均時間 | 30秒以内（導入前比75%削減） |
| **NPS** | Net Promoter Score | 50以上 |
| **Feature Adoption Rate** | 各機能の利用率 | コア機能80%以上 |

これらのメトリクスが改善すれば、ARRもARPAもチャーンレートも自然に改善する。逆に、ARRだけを追いかけると、短期的な施策に走り、長期的な顧客価値を毀損するリスクがある。

### 助言3：フライホイールを最速で回す「Minimum Viable Flywheel」を設計せよ

ベンチマークDBの完成を待つ必要はない。以下のステップで、今すぐフライホイールの回転を始められる。

1. **足場業界パック（MS1）の10社から始めよ。** この10社の契約データを匿名化し、「足場業界の契約条件ベンチマーク（Beta版）」として、同業他社に見せよ。
2. **「このデータが見たければConPassに加入せよ」** というシンプルなバリュープロポジションで、次の20社を獲得せよ。
3. **30社になったらデータの価値が飛躍的に高まり、次の50社は向こうから来る。**

これが「Minimum Viable Flywheel」だ。254社すべてが揃うまで待つ必要はない。10社で始めよ。

### 助言4：ARPA引き上げは「価格改定」ではなく「価値の階段」で設計せよ

ARPA 50万円→100万円の道筋として料金改定を計画しているが、**顧客は価格改定には抵抗する。価値の増加には喜んで支払う。**

具体的に：

- **Step 1（月額+1万円）：** 「御社の契約更新漏れを自動検知し、年間○万円の損失を防ぎました」と、実際に防いだ損失額を定量表示する。この防止額がオプション料金の10倍以上であれば、顧客は喜んで払う
- **Step 2（月額+2万円）：** 「御社の支払条件は業界平均より15日長い（不利）。ベンチマークに基づく交渉で○万円/年のキャッシュフロー改善が可能」と、具体的な改善余地を示す
- **Step 3（月額+3万円）：** 取引先の信用リスクが変化した瞬間にアラートを出す。「取引先Aの信用スコアが低下。未回収リスク○万円。推奨アクション：支払条件の見直し」

**各ステップで、支払い金額の10倍以上のROIを実感させよ。** そうすれば、顧客は自ら上位プランを求める。

### 助言5：開発体制をDay 1の組織にせよ

外部委託主体の開発体制は、Day 2の兆候だ。判断が遅くなり、顧客のフィードバックループが長くなり、イノベーションのスピードが落ちる。

**56期を待たず、55期中にCTOクラスの正社員エンジニアを1名採用せよ。** この1名が、プロダクトのビジョンを深く理解し、顧客と直接対話し、技術的な意思決定を自律的に行える人物であること。BrainStation-23との協業は継続するが、技術的なリーダーシップは社内に置くべきだ。

Two-pizza teamは外部委託チームでは成立しない。製品を愛し、顧客を理解し、夜中に問題が起きたら飛び起きるチームが必要だ。

### 助言6：「削減した時間」ではなく「取り戻した時間で何ができるか」を語れ

計画書には「年間360時間の業務削減」という効果試算がある。だが、360時間の削減は顧客の心を動かさない。

**「360時間あれば、新規取引先を12社開拓できる」** と語れ。

**「360時間あれば、社長が現場に出る時間が月30時間増える」** と語れ。

中小企業の経営者にとって、時間は単なるコスト削減ではない。それは**機会**だ。ConPassが取り戻す時間で何ができるかを語ることで、ROIの議論は財務の話から経営戦略の話に変わる。

## 6. 「もしAmazonがこの市場に参入するなら」

最後に、私がもしAmazonとしてこの市場に参入するなら、どう攻めるかを率直に述べる。これはConPassへの脅威分析であると同時に、ConPassが取るべき戦略の示唆でもある。

### Amazonの参入戦略

**Step 1：AWS上に「Amazon Contract Management」をローンチする。**

- 月額$29.99（約4,500円）から。ConPassの1/6の価格
- AWS既存顧客（日本だけで数万社）に即座にアプローチ
- AI-OCRはAmazon Textract、LLMはAmazon Bedrockで実装。AIのコストは限界的にゼロに近い
- 日本語対応は翌年に完了

**Step 2：Amazon Businessとの統合。**

- Amazon Businessの購買データと契約データを突合し、「発注と契約の自動照合」を無料で提供
- これだけで、ConPassの計画する「3点照合」機能を無料でコモディティ化できる

**Step 3：データ量で圧倒する。**

- AWS上の数万社の契約データから、業界ベンチマークを数ヶ月で構築
- ConPassが5年かけて構築する254社のベンチマークDBを、半年で10倍の規模で超える

### ConPassへの示唆

上記のシナリオは、ConPassにとって最も恐ろしい脅威シナリオだ。では、このシナリオに対してConPassが取れる防衛策は何か。

**Amazonが絶対にやらないこと（やれないこと）をやれ。**

1. **物理的なオペレーション。** Amazonは倉庫を持っているが、日本全国の中小企業のオフィスに営業マンが訪問し、紙の契約書を回収し、スキャンし、原本を保管するオペレーションは、Amazonの事業モデルに合わない。**BPO統合は、Amazonが参入しても模倣できない唯一のMoat**だ。

2. **業界の深い商慣習への対応。** 足場業界のリース契約、不動産の重要事項説明書、介護の報酬改定対応。これらは業界ごとの深い知識が必要で、グローバルプラットフォームでは対応しない。**バーティカル特化を加速せよ。**

3. **士業との信頼関係。** 税理士や弁護士は、Amazonのプラットフォームよりも、日本企業が運営するサービスを信頼する可能性が高い。**士業チャネルの囲い込みは、Amazon対策としても極めて有効**だ。

4. **中小企業への寄り添い。** 30名の会社の総務担当者は、AWSのコンソールにログインしたくない。電話で質問したい。担当者の名前を覚えていてほしい。**この「人間的なタッチ」は、テクノロジー企業が最も苦手とすること**であり、日本の中小企業向けSaaSにとっての最大の武器だ。

### 結論

もしAmazonが参入するなら、テクノロジーと価格で正面から戦ってはならない。BPO統合、バーティカル特化、士業チャネル、そして中小企業への人間的な寄り添い。これらの要素を、フライホイールの中核に据えよ。

テクノロジーは道具であり、差別化の本質ではない。**顧客との関係の深さが、すべてのMoatの中で最も強固なMoat**だ。

## 最後に

この計画は「悪い計画」ではない。むしろ、分析力と構想力においては水準以上だ。だが、Amazonを創業した一人の人間として言わせてもらえば、**この計画には魂がない。**

魂とは何か。それは「この顧客のこの問題を、我々が解決しなければならない」という切迫感だ。市場規模や競合分析ではなく、一人の顧客の「困っている」という声から、すべてが始まるべきだ。

254社でいい。その254社の一社一社が、「ConPassなしでは仕事にならない」と感じるプロダクトを作れ。そうすれば、その先の2,540社も25,400社も、自然とついてくる。

**It's always Day 1.**
